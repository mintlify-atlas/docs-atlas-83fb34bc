The device configuration file (`devices.yaml`) defines all supported AI accelerator chips, their hardware identifiers, runtime images, and sandbox configurations.

## Configuration Structure

```yaml
version: "1.0"

vendors:
  - vendor_name: Huawei
    vendor_id: "0x19e5"
    chip_models:
      - config_key: ascend-910b
        model_name: Ascend 910B
        device_id: "0xd802"
        # ... additional configuration
```

## Vendor Configuration

### Vendor Fields

<AccordionGroup>
  <Accordion title="vendor_name (required)">
    **Type**: String  
    **Description**: Human-readable vendor name
    
    Examples: `Huawei`, `Metax`, `Baidu`, `Cambricon`
    
    ```yaml
    vendor_name: Huawei
    ```
  </Accordion>

  <Accordion title="vendor_id (required)">
    **Type**: String (hex)  
    **Description**: PCIe vendor identifier (16-bit hex value)
    
    Find using: `lspci -nn | grep -i accelerator`
    
    ```yaml
    vendor_id: "0x19e5"  # Huawei
    ```
    
    <Warning>Must be a valid hex string starting with `0x`</Warning>
  </Accordion>

  <Accordion title="chip_models (required)">
    **Type**: Array  
    **Description**: List of chip models from this vendor
    
    Must contain at least one chip model.
    
    ```yaml
    chip_models:
      - config_key: ascend-910b
        # ... chip model configuration
    ```
  </Accordion>
</AccordionGroup>

## Chip Model Configuration

### Basic Fields

<AccordionGroup>
  <Accordion title="config_key (required)">
    **Type**: String  
    **Description**: Unique identifier used throughout XW CLI
    
    This key maps to:
    - Runtime parameter templates
    - Model compatibility definitions
    - Device allocation requests
    
    **Format**: Lowercase, hyphen-separated
    
    ```yaml
    config_key: ascend-910b
    ```
    
    <Warning>Must be globally unique across all vendors</Warning>
  </Accordion>

  <Accordion title="model_name (required)">
    **Type**: String  
    **Description**: Human-readable chip model name
    
    Used in UI displays and logs.
    
    ```yaml
    model_name: Ascend 910B
    ```
  </Accordion>

  <Accordion title="device_id (required)">
    **Type**: String (hex)  
    **Description**: PCIe device identifier (16-bit hex value)
    
    Find using: `lspci -nn -d {vendor_id}:`
    
    ```yaml
    device_id: "0xd802"  # Ascend 910B
    ```
  </Accordion>

  <Accordion title="generation (optional)">
    **Type**: String  
    **Description**: Chip generation grouping
    
    Used for categorizing related chip models.
    
    ```yaml
    generation: Ascend 9xx
    ```
  </Accordion>

  <Accordion title="capabilities (optional)">
    **Type**: Array of strings  
    **Description**: Supported features and precision modes
    
    ```yaml
    capabilities:
      - int8
      - fp16
      - inference
    ```
  </Accordion>

  <Accordion title="chips_per_device (optional)">
    **Type**: Integer  
    **Default**: 1
    
    **Description**: Number of AI chips per physical PCI device
    
    Used for multi-chip cards (e.g., dual-chip NPUs).
    
    ```yaml
    chips_per_device: 2  # Dual-chip card
    ```
    
    **Example**: Ascend 310P (2 chips per card)
    ```yaml
    - config_key: ascend-310p
      model_name: Ascend 310P
      device_id: "0xd500"
      chips_per_device: 2
    ```
  </Accordion>
</AccordionGroup>

## Chip Variants

Variants distinguish between sub-versions of the same chip model that share the same `device_id` but have different `subsystem_device_id`.

### Variant Configuration

```yaml
chip_models:
  - config_key: ascend-910b
    model_name: Ascend 910B
    device_id: "0xd802"
    variants:
      - subsystem_device_id: "0x3001"
        variant_key: "ascend-910b1"
        variant_name: "910B1"
      - subsystem_device_id: "0x4000"
        variant_key: "ascend-910b4"
        variant_name: "910B4"
    # Shared by all variants
    runtime_images:
      vllm:
        arm64: harbor.tsingmao.com/xw-cli/vllm-ascend:v0.14.0rc1-arm64
```

### Variant Fields

<AccordionGroup>
  <Accordion title="subsystem_device_id (required)">
    **Type**: String (hex)  
    **Description**: PCIe subsystem device identifier
    
    Find using:
    ```bash
    cat /sys/bus/pci/devices/0000:XX:YY.Z/subsystem_device
    ```
    
    ```yaml
    subsystem_device_id: "0x3001"
    ```
  </Accordion>

  <Accordion title="variant_key (required)">
    **Type**: String  
    **Description**: Unique identifier for this variant
    
    Used as `config_key` when this variant is matched.
    
    ```yaml
    variant_key: "ascend-910b1"
    ```
  </Accordion>

  <Accordion title="variant_name (optional)">
    **Type**: String  
    **Description**: Human-readable variant name
    
    ```yaml
    variant_name: "910B1"
    ```
  </Accordion>
</AccordionGroup>

### Matching Logic

Three-phase matching for device detection:

1. **Phase 1**: If model has variants, try to match `subsystem_device_id`
   - Returns base model + matched variant
   
2. **Phase 2**: Try exact match with legacy `subsystem_device_id` field (deprecated)
   - For backward compatibility only
   
3. **Phase 3**: Use base model config as fallback
   - When no variant matches or subsystem_device_id is unavailable

<Info>
  All variants share the same `runtime_images` from the base model configuration.
</Info>

## Hardware Topology

Define physical chip interconnection topology for optimal device allocation.

### Topology Configuration

```yaml
topology:
  boxes:
    - devices: [0, 1, 2, 3]  # Group 1: High-speed interconnect
    - devices: [4, 5, 6, 7]  # Group 2: High-speed interconnect
```

### Topology Fields

<AccordionGroup>
  <Accordion title="boxes (required)">
    **Type**: Array of device groups  
    **Description**: List of high-speed interconnected device groups
    
    **Distance Calculation**:
    - Same box: distance = 0
    - Different boxes: distance = |box_index_a - box_index_b|
    
    ```yaml
    topology:
      boxes:
        - devices: [0, 1, 2, 3]  # Box 0
        - devices: [4, 5, 6, 7]  # Box 1
    # Distance between chip 0 and chip 1: 0 (same box)
    # Distance between chip 0 and chip 4: 1 (different boxes)
    ```
  </Accordion>

  <Accordion title="devices (required)">
    **Type**: Array of integers  
    **Description**: Logical chip indices in this topology box
    
    Use logical chip indices (as shown in `xw device list`), not physical device indices.
    
    ```yaml
    devices: [0, 1, 2, 3]
    ```
  </Accordion>
</AccordionGroup>

### Example: Ascend 910B Topology

8 cards in 2 groups of 4-card HCCS interconnect:

```yaml
- config_key: ascend-910b
  model_name: Ascend 910B
  device_id: "0xd802"
  topology:
    boxes:
      - devices: [0, 1, 2, 3]  # Cards 0-3 interconnected
      - devices: [4, 5, 6, 7]  # Cards 4-7 interconnected
```

### Example: Ascend 310P Topology

Dual-chip cards (4 physical cards = 8 logical chips):

```yaml
- config_key: ascend-310p
  model_name: Ascend 310P
  device_id: "0xd500"
  chips_per_device: 2
  topology:
    boxes:
      - devices: [0, 1]  # Physical card 0
      - devices: [2, 3]  # Physical card 1
      - devices: [4, 5]  # Physical card 2
      - devices: [6, 7]  # Physical card 3
```

## Runtime Images

Define Docker images for inference engines by CPU architecture.

### Runtime Images Structure

```yaml
runtime_images:
  vllm:
    arm64: harbor.tsingmao.com/xw-cli/vllm-ascend:v0.14.0rc1-arm64
    amd64: NONE
  mindie:
    arm64: harbor.tsingmao.com/xw-cli/mindie:2.2.RC1-800I-A2-py311-openeuler24.03-lts-arm64
    amd64: NONE
  omni-infer:
    arm64: omniinfer-xw:latest
    amd64: NONE
```

### Architecture Support

<Tabs>
  <Tab title="arm64">
    **Platform**: ARM 64-bit (aarch64)
    
    Common for:
    - Huawei Kunpeng servers
    - ARM-based cloud instances
    - Embedded AI systems
    
    ```yaml
    arm64: harbor.tsingmao.com/xw-cli/vllm-ascend:v0.14.0rc1-arm64
    ```
  </Tab>
  
  <Tab title="amd64">
    **Platform**: x86 64-bit (x86_64)
    
    Common for:
    - Intel/AMD servers
    - Standard cloud instances
    - Desktop workstations
    
    ```yaml
    amd64: harbor.tsingmao.com/xw-cli/vllm-metax:0.12.0-amd64
    ```
  </Tab>
  
  <Tab title="NONE">
    **Not Supported**: Architecture not available
    
    Use `NONE` when an engine is not supported on an architecture.
    
    ```yaml
    amd64: NONE  # Not available on x86
    ```
  </Tab>
</Tabs>

### Supported Engines

- **vllm**: vLLM inference engine
- **mindie**: MindIE inference engine  
- **omni-infer**: Omni-Infer engine
- **mlguider**: MLGuider inference engine

<Info>
  The system automatically selects the image matching the host architecture at runtime.
</Info>

## Extended Sandboxes

Define containerization requirements for inference engines without code changes.

### Sandbox Structure

```yaml
ext_sandboxes:
  # Common configuration (shared by all engines)
  devices:
    - /dev/davinci0
    - /dev/davinci1
    - /dev/davinci_manager  # Shared device
  volumes:
    - /usr/local/Ascend/driver/lib64/:/usr/local/Ascend/driver/lib64/
    - /root/.cache:/root/.cache
  runtime: runc
  
  # Engine-specific configurations
  vllm:
    device_env: ASCEND_RT_VISIBLE_DEVICES
    environment:
      ASCEND_VISIBLE_DEVICES: "0,1,2,3,4,5,6,7"
      ASCEND_SLOG_PRINT_TO_STDOUT: "1"
    privileged: true
    shm_size_gb: 100
    capabilities:
      - SYS_ADMIN
      - SYS_RAWIO
      - IPC_LOCK
```

### Common Configuration

<AccordionGroup>
  <Accordion title="devices (shared)">
    **Type**: Array of strings  
    **Description**: Device node paths to mount
    
    **Mounting Behavior**:
    - Paths ending with digit (e.g., `/dev/davinci0`): Per-device mounting
    - Paths without digit (e.g., `/dev/davinci_manager`): Shared across containers
    
    ```yaml
    devices:
      - /dev/davinci0          # Mounted only when device 0 allocated
      - /dev/davinci1          # Mounted only when device 1 allocated
      - /dev/davinci_manager   # Always mounted (shared)
    ```
  </Accordion>

  <Accordion title="volumes (shared)">
    **Type**: Array of strings  
    **Format**: `host:container` or `path` (same for both)
    
    **Description**: Volume mounts for libraries and caches
    
    ```yaml
    volumes:
      - /usr/local/Ascend/driver/lib64/:/usr/local/Ascend/driver/lib64/
      - /root/.cache:/root/.cache
    ```
  </Accordion>

  <Accordion title="runtime (shared)">
    **Type**: String  
    **Default**: `runc`
    
    **Description**: Docker runtime to use
    
    Common values: `runc`, `nvidia`
    
    ```yaml
    runtime: runc
    ```
  </Accordion>
</AccordionGroup>

### Engine-Specific Configuration

<AccordionGroup>
  <Accordion title="device_env (required)">
    **Type**: String  
    **Description**: Environment variable for device visibility
    
    Automatically set to comma-separated device indices (e.g., `0,1,2`).
    
    **Common Values**:
    - Ascend NPU: `ASCEND_RT_VISIBLE_DEVICES`
    - NVIDIA GPU: `CUDA_VISIBLE_DEVICES`
    - Metax: `CUDA_VISIBLE_DEVICES`
    - Kunlun XPU: `XPU_VISIBLE_DEVICES`
    
    ```yaml
    device_env: ASCEND_RT_VISIBLE_DEVICES
    ```
  </Accordion>

  <Accordion title="environment (optional)">
    **Type**: Map of key-value pairs  
    **Description**: Additional static environment variables
    
    ```yaml
    environment:
      ASCEND_VISIBLE_DEVICES: "0,1,2,3,4,5,6,7"
      ASCEND_SLOG_PRINT_TO_STDOUT: "1"
      ASCEND_GLOBAL_LOG_LEVEL: "3"
    ```
  </Accordion>

  <Accordion title="privileged (optional)">
    **Type**: Boolean  
    **Default**: false
    
    **Description**: Enable privileged container mode
    
    Required by some accelerators for device access.
    
    ```yaml
    privileged: true
    ```
    
    <Warning>Privileged mode reduces container isolation</Warning>
  </Accordion>

  <Accordion title="shm_size_gb (optional)">
    **Type**: Integer  
    **Default**: 16  
    **Unit**: Gigabytes
    
    **Description**: Shared memory size for IPC
    
    Required for:
    - Inter-process communication in distributed inference
    - PyTorch DataLoader workers
    - Tensor sharing between processes
    
    ```yaml
    shm_size_gb: 100
    ```
  </Accordion>

  <Accordion title="capabilities (optional)">
    **Type**: Array of strings  
    **Description**: Linux capabilities required
    
    **Common Capabilities**:
    - `SYS_ADMIN`: System administration operations
    - `SYS_RAWIO`: Direct device I/O access
    - `IPC_LOCK`: Memory locking for device buffers
    - `SYS_RESOURCE`: Resource limit adjustments
    - `NET_ADMIN`: Network administration (for multi-node)
    
    ```yaml
    capabilities:
      - SYS_ADMIN
      - SYS_RAWIO
      - IPC_LOCK
      - SYS_RESOURCE
    ```
  </Accordion>
</AccordionGroup>

### Merge Strategy

Common and engine-specific configurations are merged:

- **Devices**: `common.devices + engine.devices` (appended)
- **Volumes**: `common.volumes + engine.volumes` (appended)
- **Runtime**: Engine overrides common if specified
- **Other fields**: Engine-specific only

## Complete Example: Ascend 910B

```yaml
vendors:
  - vendor_name: Huawei
    vendor_id: "0x19e5"
    
    chip_models:
      - config_key: ascend-910b

        # Runtime images
        runtime_images:
          vllm:
            arm64: harbor.tsingmao.com/xw-cli/vllm-ascend:v0.14.0rc1-arm64
            amd64: NONE
          mindie:
            arm64: harbor.tsingmao.com/xw-cli/mindie:2.2.RC1-800I-A2-py311-openeuler24.03-lts-arm64

        # Extended sandboxes
        ext_sandboxes:
          # Common configuration
          devices:

            - /dev/davinci2
            - /dev/davinci3

            - /usr/local/dcmi:/usr/local/dcmi

          vllm:
            device_env: ASCEND_RT_VISIBLE_DEVICES
```

## Finding Hardware IDs

### PCIe Vendor ID

```bash
lspci -nn | grep -i accelerator
# Output: 01:00.0 Processing accelerators [1200]: Huawei [19e5]:d802
#                                                           ^^^^- vendor_id
```

### PCIe Device ID

```bash
lspci -nn -d 0x19e5:
# Output: 01:00.0 Processing accelerators [1200]: Huawei [19e5]:d802
#                                                                ^^^^- device_id
```

### Subsystem Device ID

```bash
# Find PCI address first
lspci | grep -i accelerator
# Output: 01:00.0 Processing accelerators: Huawei

# Read subsystem device ID
cat /sys/bus/pci/devices/0000:01:00.0/subsystem_device
# Output: 0x3001
```

## Validation

Configuration validation checks:

<Check>Version field is present</Check>
<Check>At least one vendor defined</Check>
<Check>Vendor IDs are valid hex strings</Check>

<Check>No duplicate config_key values</Check>

## Best Practices

<Check>Use descriptive config_key names (e.g., `ascend-910b`, not `a910b`)</Check>
<Check>Document custom configurations with YAML comments</Check>
<Check>Test device detection with `xw device list` after changes</Check>
<Check>Keep vendor IDs and device IDs in hex format with `0x` prefix</Check>
<Check>Define topology for multi-device systems</Check>
<Check>Specify both arm64 and amd64 images when available</Check>

<Warning>
  Device configuration changes require a server restart to take effect.
</Warning>
